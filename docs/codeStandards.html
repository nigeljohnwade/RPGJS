Code Standards - Javascript

This section defines the code standards that I have attempted to adhere to on this project

Object Definitions

If an object is static, i.e. it does not need instantiating before use, then the following pattern is used:

                
                    var object = {
                        method1: function(){
                        
                        }
                        ,method2: function(){
                        
                        }
                    }

Note the use of commas at the beginning of a method definition not after the curly brace. this makes it easier to comment out or delete a method in its entireity

If an object is to be instantiated it uses the following pattern:

                
                    function object(){
                    
                        //private properties
                        var property1, property2;
                        
                        //public properties
                        this.property1;
                        this.property2;
                    }
                    //public methods
                    object.prototype.method1 = function(){
                        //statements
                    }

Using prototype attachment for methods requires less browser overheads than defining them inside the object like so:

                
                    function object(){
                    
                        //private properties
                        var property1, property2;
                        
                        //public properties
                        this.property1;
                        this.property2;
                        
                        //inefficient public methods
                        this.method1 = function(){
                        //statements
                        }
                        
                    }

Public methods defined inside the object are instantiated along with each object, prototype attached methods are not instantiated with the object, only a reference to them is created. This creates less browser overhead.

Semi-colons

Always use a semi-colon at the end of a line, this makes it easier to read as it shows that a line has intentionally been ended rather than accidentally being ended.

Code Standards - HTML

This section defines the HTML code standards that I have attempted to adhere to on this project

Case

I use lower case for tags and lower case for attributes 

                
                    <h1 class="heading">Some Heading</h1>

Attribute are always double quoted in raw HTML code, in server side code (PHP) or client side scripting (JavaScript) languages feel free to not use quotes in order to reduce excessive escape characters

Images

The alt attribute for images is always included, if the image is decorative use "*" rather than "".

Images should have their width and height supplied wherever possible so that the browser display does not need redrawing after images laod

General

The following are some general things that don't fir specifically any where else

Never use a <br /> to create vertical space, vertical space is presentational and goes in the CSS.

Never use &nbsp; to create horizontal space, horizontal space is presentational and goes in the CSS.

Self Closing Tags

Always use a self closing verison of a tag if it is avaiable.
